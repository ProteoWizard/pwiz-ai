## Context / why

Skyline has a well-designed declarative command-line system in
`pwiz_tools/Skyline/CommandArgs.cs` (~3540 lines): each argument is a descriptor with an embedded
`ProcessValue` lambda, arguments are grouped, and help is generated by enumerating the declarations
and rendering ascii / unicode / HTML via `ConsoleTable`. The *generic* machinery is cleanly
separable from Skyline's *domain-specific* declarations. This sprint lifts that generic machinery
into a new portable DLL so a second tool (OspreySharp) can reuse it instead of reinventing it, and
so the design has a single shared home. `CommonUtil` is net472-only, non-SDK, and GUI/web-laden —
not worth fully modernizing — so the new `PortableUtil` is a clean SDK multi-target leaf that
portable pieces migrate into over time.

## What to build

### 1. New project `pwiz_tools/Shared/PortableUtil/PortableUtil.csproj`
- SDK-style, `<TargetFrameworks>net472;net8.0</TargetFrameworks>`, `<Platforms>AnyCPU;x64</Platforms>`,
  `Nullable disable`, `ImplicitUsings disable`. **AssemblyName** `pwiz.PortableUtil`,
  **RootNamespace** `pwiz.Common`. **Pure-BCL leaf** — must NOT reference net472-only CommonUtil.
- Inline the MSBuild props in the csproj (do **not** add a `Directory.Build.props` under `Shared/` —
  it would leak into sibling non-SDK projects).
- Framework types go in namespace **`pwiz.Common.CommandLine`** (keeps the `pwiz.Common.*` root so
  consumers' usings read naturally; only the assembly is new).

### 2. Move `ConsoleTable` only — keep `Assume` in CommonUtil (no type-forwarding)
- Move `pwiz_tools/Skyline/Util/ConsoleTable.cs` → PortableUtil `pwiz.Common.CommandLine`, make it
  `public` (was `internal`). Single consumer (`CommandArgs.cs`) — update its `using`.
- ConsoleTable's ONLY non-BCL dependency is `Assume.IsTrue(widths.All(w => w > 0))` (ConsoleTable.cs:319).
  **Replace inline** with `if (!widths.All(w => w > 0)) throw new InvalidOperationException(...)`.
  This preserves intent: `ExceptionUtil.IsProgrammingDefect` (Skyline/Util/Util.cs:1898) returns
  `true` for `InvalidOperationException` (it only excludes InvalidData/IO/OperationCanceled/
  UnauthorizedAccess/UserMessage), so it's still classified as a coding defect like the old assertion.
- **`Assume` STAYS in CommonUtil** — no move, no `[TypeForwardedTo]`, no CommonUtil→PortableUtil
  reference. One `Assume.IsTrue` call isn't worth dragging `Assume` into PortableUtil, and this leaves
  PortableUtil a pure-BCL leaf.
- During extraction, **audit the moved framework types** for any other `pwiz.Common` helper calls
  (`Assume`, `TextUtil`, …) and inline/replace them likewise, so PortableUtil compiles with zero pwiz
  references (the known non-BCL touchpoints — resources, IsRemoteUrl, HtmlEncode — are seamed in step 4).

### 3. Extract + generify the framework (out of `CommandArgs.cs`)
- Introduce a non-generic **`ArgumentBase`** (Name, ArgumentText, ValueExample, Values, OptionalValue,
  HasValueChecking, InternalUse, Description) and **`Argument<TContext> : ArgumentBase`** carrying
  `Func<TContext,NameValuePair,bool> ProcessValue`. Keep **`NameValuePair` non-generic** (references
  only `ArgumentBase`) so it isn't virally parameterized.
- Move into PortableUtil: `Argument<TContext>`, `NameValuePair`, `ArgumentGroup<TContext> : IUsageBlock`,
  `IUsageBlock`, `ParaUsageBlock`, base `UsageException`, and the ~10 context-free value exceptions
  (`ValueMissing/Unexpected/Invalid/InvalidBool/Int/Double/Date/Path/OutOfRangeInt/OutOfRangeDouble`).
  Make these value exceptions take a **pre-formatted (already-localized) message** so PortableUtil
  carries no `.resx`.
- **Stays in Skyline:** the `CommandArgs` controller, all `ARG_*`/`GROUP_*` declarations,
  `DocArgument`/`RefineArgument`/`ToolArgument` (re-based onto `Argument<CommandArgs>`), `Usage()`,
  `GenerateUsageHtml()` (so the HTML stylesheet via `DocumentationGenerator.GetStyleSheetHtml()` stays
  Skyline-side — no stylesheet param crosses into PortableUtil), and the Skyline-domain exceptions
  (charge/iontype/mod/etc.) subclassing the moved bases.

### 4. Break the two Skyline couplings via seams (in PortableUtil)
- **Descriptions + headers:** `IArgDescriptionProvider { string GetDescription(string argName);
  string AppliesToHeader/ArgumentHeader/DescriptionHeader; }`. `ArgumentBase.Description` calls the
  provider. Skyline supplies one backed by `CommandArgUsage.ResourceManager` (key
  `"_" + name.Replace('-','_')`) + the `CommandArgGroup_ToString_*` header strings.
- **IsRemoteUrl:** `NameValuePair.ValueFullPath` currently calls `CommandArgs.IsRemoteUrl`. Add a
  static seam (e.g. `ArgUsage.IsRemoteUrl = Func<string,bool>`, default `_ => false`); Skyline sets it
  to `CommandArgs.IsRemoteUrl`.
- **HtmlEncode:** the moved `ToHtmlString` uses `HttpUtility.HtmlEncode` (net472 only). Prefer
  `System.Net.WebUtility.HtmlEncode` (BCL on both targets) **but verify byte-identical output vs the
  committed goldens**; keep `HttpUtility` if it differs.

### 5. Low-churn Skyline adoption of the moved types
- File-scoped `using` aliases at the top of `CommandArgs.cs`:
  `using Argument = pwiz.Common.CommandLine.Argument<pwiz.Skyline.CommandArgs>;` (and `ArgumentGroup`,
  `NameValuePair`, `IUsageBlock`, `UsageException`). The 100+ `new DocArgument(...)` / `Argument` field
  declarations then compile unchanged.
- Keep thin **nested re-export shims** in `CommandArgs`
  (`public class ArgumentGroup : pwiz.Common.CommandLine.ArgumentGroup<CommandArgs> {…}`, same for
  `Argument`) so external references like `CommandArgs.ArgumentGroup` in the tests still resolve with
  zero test edits.
- Add `ProjectReference` Skyline.csproj → PortableUtil; add PortableUtil to `Skyline.sln` with config
  rows mirroring CommonUtil. **Skyline.sln has only x64/x86 configs (no AnyCPU)** — map the sln's x86
  config to PortableUtil's **AnyCPU**.
